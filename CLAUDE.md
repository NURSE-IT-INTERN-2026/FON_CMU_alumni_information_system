## Architecture

### Core Stack

- **Next.js 16.2** with App Router (`app/` directory). This version has breaking changes from earlier Next.js ŌĆö read the relevant guide in `node_modules/next/dist/docs/` before writing code.
- **React 19.2** with server and client components.
- **Prisma 7** ORM with PostgreSQL. Schema at `prisma/schema.prisma`. Client generates to `app/generated/prisma/` (import from there, not `@prisma/client`).
- **Prisma PostgreSQL adapter** (`@prisma/adapter-pg`) ŌĆö the Prisma client is initialized with the `PrismaPg` adapter in `lib/prisma.ts`, using `DATABASE_URL` from `.env`.
- **Tailwind CSS 4** with PostCSS plugin (`@tailwindcss/postcss`). CSS-first configuration in `app/globals.css` ŌĆö no `tailwind.config.js`. Uses `@theme inline` block for custom theme colors (primary: `#5b21b6` purple, accent: `#e8a838` gold).
- **Vitest 4** for testing. Config at `vitest.config.ts`. Tests in `tests/`.
- **Path alias:** `@/*` maps to project root.

### Prisma Client Import Pattern

```ts
import prisma from "@/lib/prisma";
```

The singleton pattern in `lib/prisma.ts` prevents multiple client instances during hot reload. The client is imported from `../app/generated/prisma/client`, **not** `@prisma/client`.

### Auth & Roles

- **Session-based auth** using `bcryptjs` + HTTP-only cookies (`fon-cmu-session`). Session tokens stored in `Session` model, 7-day expiry.
- **CMU OAuth2** with PKCE via Microsoft Entra ID (`lib/oauth.ts`). Callback at `/api/auth/callback/` (route `app/api/auth/callback/route.ts`; the registered redirect URI is the `CALLBACK_URL` env var, which both the authorize request and the token exchange read).
- **2 Roles:** `superadmin` (full CRUD + user management), `admin` (CRUD + import/export).
- **Alumni portal** (`/graduates/*`): separate self-service session flow ŌĆö email/password login, **email-verification + admin-approval sign-up** (a signup creates an `UNVERIFIED` account and emails a verification link; clicking it flips to `PENDING`; an admin must then approve on the manage-accounts table before the alumni can log in), first-login TOS acceptance. Reuses the `Session` model with `sessionType: ALUMNI`; auth routes under `app/api/alumni-auth/`.
- **Role context:** `lib/role-context.tsx` provides `useRole()`, `useCanWrite()`, `useIsAdmin()` hooks.
- **Write permission check:** `lib/permissions.ts` ŌĆö `checkWritePermission()` returns 401/403 for unauthorized requests.
- **Rate limiting:** `lib/rate-limit.ts` ŌĆö in-memory sliding-window (5 attempts / 15 min).
- **Auth middleware via `proxy.ts`** (Next.js 16 renamed middleware to proxy) ŌĆö enforces CSP headers and redirects unauthenticated users. Also enforced at the layout level: `app/(admin)/layout.tsx` guards the admin area; `app/graduates/(authed)/layout.tsx` guards the alumni portal.

### API Route Pattern

Each data entity follows a consistent route structure:
- `GET /api/{entity}` ŌĆö list (paginated, searchable)
- `POST /api/{entity}` ŌĆö create
- `GET/PUT/DELETE /api/{entity}/[id]` ŌĆö read/update/delete
- `POST /api/{entity}/import` ŌĆö Excel import
- `GET /api/{entity}/export` ŌĆö Excel export
- `POST /api/{entity}/bulk-delete` ŌĆö bulk delete by IDs
- `POST /api/alumni/create-with-related` + `PUT /api/alumni/update-with-related/[id]` ŌĆö create/update an alumni together with related records (full-form; one save can affect other pages)
- DELETE is a **soft delete**; recovery is superadmin-only via `POST /api/trash/restore` and `POST /api/trash/hard-delete`
- Every mutating route must call `checkWritePermission` (`@/lib/permissions`) and `logActivity` (`@/lib/activity-log`)
- **Standard CRUD + import/export/bulk-delete entities:** alumni, alumni-agency, associations, awards, graduate-committee, model-representatives, potentials.
- **Deviations from the standard pattern:**
  - `alumni-activity` ŌĆö read-only engagement analytics (login activity + accounts); no `/[id]`/import/export/bulk-delete. Session-guarded + 60s-TTL-cached (`withTtlCache("alumni-activity")`), same class as `dashboard`/`alumni-count`. Login counts read `ActivityLog` where `action='LOGIN'` + `resource='alumni_auth'` + `actorType='ALUMNI'` (see Known Pitfalls).
  - `news` ŌĆö no `import`/`export`; DELETE ŌåÆ `status: DISCONTINUED` (NOT a soft delete, NOT trash-recoverable). **Pinned news ("ÓĖøÓĖŻÓĖ░ÓĖŖÓĖ▓ÓĖ¬ÓĖ▒ÓĖĪÓĖ×ÓĖ▒ÓĖÖÓĖśÓ╣īÓĖ¬ÓĖ│ÓĖäÓĖ▒ÓĖŹ"):** `GET /api/news` **excludes pinned by default** (`pinnedAt: null`) and `?pinned=true` returns only pinned (`pinnedAt desc`, used by the top-of-page section on both the admin + alumni news pages); `POST /api/news/[id]/pin` `{ pinned }` is the admin toggle. Discontinuing (single DELETE / bulk-delete / PUT statusŌåÆDISCONTINUED) clears `pinnedAt` (pinned ŌćÆ not-discontinued).
  - `users` ŌĆö no `import`/`export`/`bulk-delete`; write ops are superadmin-only (`checkSuperAdminPermission`).
  - `alumni-accounts` ŌĆö admin alumni-account mgmt; no import/export/bulk-delete. The list filters `passwordHash != null` **and excludes `UNVERIFIED`** (`accountStatus: { not: "UNVERIFIED" }`) ŌĆö admins track only accounts that cleared email verification (PENDING/ACTIVE/REJECTED); `?status=` accepts `pending|active|rejected` (NOT `unverified`), and `UNVERIFIED` is surfaced in no admin-tracking surface (this list / the dashboard count / the alumni-activity count). Returns `accountStatus` + `signupVerification` + `rejectionReason` + `rejectedAt`. Sub-routes: `/[id]/suspend` (toggle `suspendedAt` + kill sessions), and the **admin-approval** set `/[id]/approve` (PENDING|REJECTED ŌåÆ ACTIVE; creates the deferred Education + `syncPrimarySnapshot` + approval email; clears `rejectionReason`/`rejectedAt`), `/[id]/reject` (**requires `{reason}`** ŌĆö zod non-empty; ŌåÆ REJECTED + kill sessions + stamps `rejectionReason`/`rejectedAt` + rejection email w/ the reason + a ÓĖóÓĖĘÓ╣łÓĖÖÓĖäÓĖ│ÓĖéÓĖŁÓ╣āÓĖ½ÓĖĪÓ╣ł link; re-approvable; the review modal shows the latest reason + the REJECT history from `GET /api/alumni/[id]/activity`), `/[id]/reverify` (re-fetch CMU, rebuild `signupVerification`). All write sub-routes use `checkWritePermission` (admin + superadmin) + `logActivity` (`APPROVE`/`REJECT`/`SUSPEND`/`RESTORE`). **Exception ŌĆö `/[id]/delete`** is the one **superadmin-only** sub-route (`checkSuperAdminPermission`): it deletes the alumni *account* (login credentials) while **KEEPING the data record** ŌĆö nulls `email`+`passwordHash`+`accountStatus`ŌåÆ`UNVERIFIED`+`signupVerification`+`emailVerifiedAt`+`suspendedAt`+`rejectionReason`+`rejectedAt`+`hasLoggedIn`+`lastLoginAt`+`tosAcceptedAt`, does NOT set `deletedAt` and does NOT touch `studentId`/name/education; in one `$transaction` it also `deleteMany`s the alumni's `Session`s + `EmailVerification` + `PasswordReset` tokens, then `bustCache("dashboard")`+`bustCache("alumni-activity")` + logs `DELETE`/`alumni_auth` `{ source: "admin_delete_account" }` (no email sent). Purpose: let the alumni re-sign-up to re-test the signup email ŌĆö because signup dedups on `email` first (`findFirst`+`@unique`), nulling `email`+`passwordHash` makes the re-signup take the UPDATE branch on the preserved `studentId` row and re-fire `sendEmailVerificationEmail`. The UI is a superadmin-only (`useRole()==="superadmin"`) trash icon in `AlumniAccountsTab` (`settings/users/page.tsx`) with a confirm modal explaining the record is kept + re-sign-up enabled.
  - `alumni-profile` ŌĆö no `/[id]`; operates on the logged-in alumni (`getAlumniSession`). GET/PUT/DELETE. `alumni-profile/educations` adds alumni-self GET/POST of education records.
  - `cmu-alumni` ŌĆö read-only external Registrar proxy (GET list/search) + `lookup?studentId=&alumniId=` (single-record preview for the add-education form; with `alumniId`, returns `samePersonWarning` if the record belongs to a different person).
  - `educations` ŌĆö degree records (1:N per alumni). `GET/POST /api/alumni/[id]/educations` (admin) and `GET/POST /api/alumni-profile/educations` (alumni-self) for list/add; `GET/PUT/DELETE /api/educations/[id]` for one record (admin OR owning alumni via `resolveWriter`). Every add, and a PUT that changes `studentId`, must pass `assertEducationSamePerson` (`lib/education-identity.ts`) ŌĆö a studentId whose CMU birthday differs from the alumni's is rejected with 400 (can't attach a stranger's degree). No import/export/bulk-delete.
  - `logs` ŌĆö otherwise read-only (GET list only). Adds a **superadmin-only** `POST /api/logs/bulk-delete` `{ ids }` that **hard-deletes** (`deleteMany`; `ActivityLog` has no `deletedAt`). Not soft-delete, not trash-recoverable, and **deliberately does NOT log the deletion** (logging it would defeat the purpose of removing log entries). UI gate on the logs page is `useRole() === "superadmin"` (NOT `useIsAdmin()`, which covers both roles).

### Route Correlations, Redirects & Entry Points

When adding/changing a route, keep this map honest (see Working Protocol "On touching routes").

**Correlations ŌĆö routes that pair or overlap (touching one affects the other):**

## Working Protocol

> Read at the start of every task. These are the rules for how I communicate results and keep this instruction file honest. Follow them unless the user says otherwise.

### Git workflow ŌĆö branch off, do the work, commit, then merge to `main`

Default flow for **every** task that changes code (the user wants it; it supersedes the generic "commit only when asked" rule):

1. **Branch off `main` first** ŌĆö before writing any code, create a descriptively-named branch from `main` and switch to it. Never work or commit directly on `main`.
2. **Do the task** on that branch.
3. **Commit when finished** ŌĆö once the work is done and verification (build/lint/tests) is green, commit it.
4. **Confirm in case of errors** ŌĆö if anything fails (build, lint, tests, a merge conflict, or any unexpected problem), STOP and ask the user before committing/merging instead of forcing it through. Proceed only once it's green, or the user OKs it.
5. **Merge to `main`** ŌĆö after committing, check out `main` and merge the branch back with `--no-ff` (keeps the feature unit visible in history). Leave the working tree clean and end on `main`.

Don't `git push` unless explicitly asked. If a request only needs exploration/answers (no code change), skip the flow.

### On finishing any task ŌĆö produce a completion report

Before claiming a task is done, give a short structured report (tight bullets ŌĆö never a file dump). Cover all of:

- **Plan executed** ŌĆö the goal and the approach (1ŌĆō3 lines). If the approach changed mid-task, say why.
- **Files changed** ŌĆö `created` / `updated` / `deleted`, each with its real path (e.g. `app/(admin)/management/awards/page.tsx`). Don't lump them; distinguish creates from edits from deletes.
- **Libraries / tech used** ŌĆö anything used or newly introduced (e.g. `@tanstack/react-query`, `zod`, `exceljs`). If a new dependency was added, name it + version + why.
- **Branch** ŌĆö the current git branch (e.g. `main`, `feature/x`). State whether changes are committed or still uncommitted, and the commit hash if committed. Follow the **Git workflow** above ŌĆö branch off `main` first, commit when done, merge back to `main`; pause for the user's confirmation if anything fails. Don't push unless asked.
- **Verification** ŌĆö what confirms it works: tests run (+ pass/fail), `npm run lint` / `npm run build` status, manual steps. If something was NOT verified, say so explicitly rather than implying it was. To verify **live behavior**, hit the already-running dev server on `:3000` with `curl` (see Known Pitfalls: *Never spawn a second `next dev`*) ŌĆö **never** start a second dev server for verification.

### On fixing a bug or problem ŌĆö produce a root-cause report

- **Root cause** ŌĆö the actual underlying cause, not the surface behavior. Dig past "X didn't work" to *why*.

### `docker compose up` reuses a stale cached app image ŌĆö rebuild with `--build` or the app serves old code
- **Symptom:** Docker says the app is "running on :3000" and a Next.js server responds, but the app is broken ŌĆö it serves at the root (`/login` returns 200) instead of under `basePath: "/alumni"` (`/alumni/login`), so every internal link/fetch (all written for `/alumni`) 404s/redirects wrong and the page won't load. Confirmed via curl: `/` ŌåÆ 307 `/login` (not the expected 308 ŌåÆ `/alumni`), and bare `/login` ŌåÆ 200.
- **Root cause:** `basePath` is a **build-time** constant baked into the standalone image (`output: "standalone"`). `docker compose up -d` reuses an existing image of the same name ŌĆö it does NOT rebuild from source. After pulling/merging code changes, an image built weeks earlier keeps running: `docker images` shows `fon-cmu-alumni-app:latest` created "4 weeks ago" while the container is minutes old. The host dev server (`npm run dev`) is NOT involved ŌĆö `ps` shows no `next`/`node` process; the `*:3000` listener is the container's published port owned by the Docker Desktop VM.
- **Prevention:** After any source change you want live in Docker, run `docker compose up -d --build app` (or `docker compose build app` then `up -d app`). Confirm the image is fresh (`docker images` ŌåÆ created "x minutes ago", new image id). Verify basePath with curl: `/` ŌåÆ 308 `/alumni`, `/alumni/login` ŌåÆ 200, bare `/login` ŌåÆ 404. `next.config.ts`'s `basePath` is NOT runtime-overridable, so a wrong basePath always means a stale build, never an env tweak.

### Dev container hot-reloads via `docker-compose.override.yml` (HMR) ŌĆö but only after a `--build`
- **What:** `docker compose up` (no `-f`) auto-merges `docker-compose.override.yml`, which overrides the `app` service to build the Dockerfile **`deps`** stage (full deps, incl. devDependencies) and run `next dev` with the source **bind-mounted** (`.:/app`) ŌĆö so code changes hot-reload like a normal host `npm run dev`. Anonymous volumes protect `/app/node_modules`, `/app/.next`, and `/app/app/generated` from the bind mount (container-owned). The command runs `npx prisma generate` (the `deps` image has no generated client) then `npm run dev -- -H 0.0.0.0`.
- **The catch:** the stale-image rule ABOVE still applies on the **first** switch to dev (or after any `package.json` change) ŌĆö `docker compose up` alone reuses the old standalone image, so you MUST run `docker compose up -d --build app` once. After that, plain `docker compose up` reuses the dev image and source edits reload via HMR with no rebuild. To go back to the production standalone container, deploy with `docker compose -f docker-compose.yml up -d --build app` (the `-f` skips the override).
- **Verify HMR:** the dev server logs `Ō£ō Ready in ŌĆ”ms`; editing a host source file then requesting its route logs `ŌŚŗ Compiling /<route> ŌĆ”` (on-demand recompile). Confirm the container reads live source, not a baked copy: `docker compose exec app grep -c '<recentToken>' '/app/app/(admin)/management/alumni-agency/page.tsx'` (>0).
- **New npm dependency (gotcha):** a host-side `npm install` / `npx shadcn@latest add ŌĆ”` updates `package.json` + `package-lock.json` (bind-mounted, so the container sees them) AND the *host's* `node_modules` ŌĆö but NOT the container's `node_modules` (anonymous volume). The dev server then **500s** with `Module not found: Can't resolve '<pkg>'`. A host `npm run build`/`npm run lint` still **passes** (it uses host `node_modules`) ŌĆö misleading, since the running dev server is the container. Fix: `docker compose exec app npm install` to install into the container's volume, then the route recovers. (Deps that resolve via the `radix-ui` umbrella need no extra direct dep ŌĆö only genuinely-new packages trigger this.)

### Dev container `public/uploads/` is a named volume ŌĆö host files there are NOT served
- **Symptom:** An image that clearly exists at `public/uploads/x.png` on the host 404s when fetched from the running dev container (`GET /alumni/uploads/x.png ŌåÆ 404`), while app-uploaded images serve fine. Looks like the upload/thumbnail feature is broken when it isn't.
- **Root cause:** `docker-compose.yml` mounts the named volume `uploads` at `/app/public/uploads` (to persist uploads in production). The dev override bind-mounts `.:/app` for HMR but does NOT remove that named volume, and compose **merges** (not replaces) volumes ŌĆö so the named volume **shadows** the host's `public/uploads/` inside the container. The container's `/app/public/uploads/` is the volume's own (root-owned) tree, separate from the host directory. App uploads write to `process.cwd()/public/uploads` = the volume, so they serve and persist; host-side files are invisible to the container.
- **Prevention:** When verifying uploads/thumbnails against the dev container, don't drop a file into the host `public/uploads/` and expect the container to serve it. Either upload through the app (`POST /api/upload`), or place the file inside the container: `docker cp file.png fon-cmu-alumni-app-1:/app/public/uploads/file.png` (and `docker exec ŌĆ” rm` it after). To see what the container actually has: `docker exec fon-cmu-alumni-app-1 ls /app/public/uploads/`.

### App is deployed under `basePath: "/alumni"`
- **Symptom:** Links, redirects, or fetches resolve to the wrong path in the deployed app.
- **Root cause:** `next.config.ts` sets `basePath: "/alumni"`. Route files define paths relative to it; some API helpers auto-prepend `BASE_PATH`, others need it manually.
- **Prevention:** Account for `basePath` in new links/calls. Don't hardcode absolute `/api/...` blindly ŌĆö check whether the helper already prepends.

### Shared `lib/` files must stay client-safe (split Prisma into a `-server` module)
- **Symptom:** Build error `Can't resolve 'dns'` (or another Node-only module) traced to a client component.
- **Root cause:** A client component imported a `lib/` module that transitively pulls in Prisma / the `pg` driver.
- **Prevention:** Keep shared `lib/` modules client-safe. Put Prisma-backed logic in a dedicated `*-server.ts` file (pattern in use: `lib/filter-facets.ts` + `lib/filter-facets-server.ts`). Never import `@/lib/prisma` across a `"use client"` boundary.

### API routes swallow Prisma errors into generic 500s
- **Symptom:** A data API returns a blank 500 with no useful detail.
- **Root cause:** Route handlers catch Prisma errors and return a generic message, hiding the real cause.
- **Prevention:** When debugging an API failure, probe the Prisma layer directly (run a script with `node --env-file=.env --import tsx`) rather than trusting the API response. The runtime DB is remote Prisma Postgres.

### Schema edits require a created + applied migration on remote Prisma Postgres
- **Symptom:** A model/schema change doesn't take effect, or the generated client disagrees with the live DB.
- **Root cause:** `npx prisma generate` alone is not enough ŌĆö the migration must be created AND applied to the remote DB, then the client regenerated.
- **Prevention:** After any `prisma/schema.prisma` change: create the migration, apply it to the remote DB, then `npx prisma generate`. Run DB scripts with `node --env-file=.env --import tsx`.

### All person-name entities use `prefix` + `firstName` + `lastName` (combined-name split)
- **Symptom:** A combined `fullName`/`name`/`thaiName` column needs splitting into the PRD's `prefix` + `firstName` + `lastName`.
- **What's split:** `Award` (was `recipientName`), `Potential`/`Association`/`GraduateCommittee` (were `fullName`), `ModelRepresentative` (was `name`), `AlumniAgency` (was `thaiName`; keeps `englishName` + its own `prefix`). Reuse `splitFullName()` in `lib/parse-name.ts` (title-strip + whitespace split) ŌĆö shared by the Excel import legacy-column fallback, `seed.ts`, `scripts/rebuild-*.ts`, and the backfill. The shared `{entity}FormSchema` carries optional names (full-form auto-fills from parent); `{entity}PageFormSchema` extends to required for the management page; `create-with-related` / `update-with-related` / **`alumni-profile`** routes all auto-fill `prefix/firstName/lastName` from the parent alumni ŌĆö the full-form payload carries NO name for these sections (like Award).
- **Migration mechanics (splitting a NOT NULL column):** needs TWO migrations ŌĆö (1) add the new cols **nullable** + `ALTER ŌĆ” DROP NOT NULL` on the legacy col, (2) after backfilling, `SET NOT NULL` on the new cols + `DROP COLUMN` the legacy. Author/apply non-interactively: `prisma migrate dev --name X --create-only` (sometimes refuses "non-interactive"), else `npx prisma migrate diff --from-config-datasource --to-schema prisma/schema.prisma --script -o prisma/migrations/<ts>_name/migration.sql` then `npx prisma migrate deploy`. The one-off backfill is a script run via `node --env-file=.env --import tsx` with `DRY_RUN=1`, then **deleted** (it reads dropped columns and won't compile against the final schema).
- **Drift gotcha ŌĆö ALWAYS run `npx prisma migrate status` before migrating.** An abandoned worktree may have applied a migration to the remote DB that is **missing from your local `prisma/migrations/`** ŌåÆ Prisma reports drift and offers `migrate reset` (data loss). Copy the missing migration file into your folder to reconcile; **never accept a reset** on the shared remote DB.

### Never spawn a second `next dev` to verify ŌĆö reuse the already-running server on :3000
- **Symptom:** At end-of-task verification the session froze / hung for the full tool timeout.
- **Root cause:** I ran `npm run dev` (or `next dev`) while the user's dev server already holds port 3000. Next 16 detects the collision via `.next/dev/` and **refuses to bind** ŌĆö it first tries another port (`ŌÜĀ Port 3000 is in use ŌĆ” using available port 3001 instead`), then prints `Ō©» Another next dev server is already running ŌĆ” Run kill <pid>`. Depending on stdin it either exits(1) or stalls on its spinner (`[?25h`) ŌĆö that stall is the freeze. It never produces a usable server to verify against, so the spawn was pointless either way.
- **Prevention:** A dev server is normally already running on `http://localhost:3000` (this project uses `basePath: "/alumni"`, so routes are at `/alumni/...` and APIs at `/alumni/api/...`). **First** probe it: `curl -s -o /dev/null -w "%{http_code}" --max-time 5 http://localhost:3000/alumni/login` ŌåÆ any HTTP status (e.g. 200/307) means reuse it. Verify changes by curling that server; do NOT run `npm run dev` / `next dev` a second time. Only if **nothing** is listening, start one with the Bash tool's `run_in_background: true` and poll the log for `Ready` ŌĆö **never foreground**, because a dev server runs forever and blocks the tool call indefinitely.

### CMU returns one record per degree ŌåÆ dedup-by-person is DISPLAY-LAYER ONLY
- **Symptom:** The same person shows up as multiple rows (and inflates counts) because they earned more than one FON degree ŌĆö e.g. a Bachelor's then a Doctoral. Each degree is a separate CMU record with a **distinct `student_id`** and different `level_id`.
- **Root cause:** `fetchCmuGraduates()` (`lib/cmu-registrar.ts`) returns the raw per-degree list; nothing in the upstream collapses them. Display/count surfaces were reading it verbatim, so one person counted N times.
- **Prevention:** Use `dedupeCmuGraduatesByPerson(graduates)` (`lib/alumni-verify.ts`) ŌĆö matches on `name_th`+`surname_th`+canonical birthday (NUL-joined), keeps the **highest** degree (`DOCTORAL>MASTER>BACHELOR>ASSOCIATE>NURSING_ASSISTANT`), and leaves incomplete-identity records verbatim. **Apply it ONLY in the table + facet display surfaces**: `/api/cmu-alumni` (on the full list, *before* search/facets/sort/pagination so records on different pages still collapse) and `lib/filter-facets-server.ts` (`getAlumniFacetValues`, so facet counts match the table). The dashboard **count** routes (`/api/alumni-count`, `/api/dashboard`) instead use the merged person count below ŌĆö NOT this dedup. **NEVER** apply it in studentId-keyed consumers ŌĆö `lib/ensure-alumni.ts` (the import studentIdŌåÆrecord map) and `app/api/alumni/[id]/route.ts` (profile lookup by studentId) MUST keep the raw `fetchCmuGraduates()`, because dedup drops each person's lower-degree `student_id`s and would silently break CMU enrichment on import / profile view. `fetchCmuGraduates()` itself stays un-deduped (its 5-min cache serves both the deduped and the raw consumers).

### Running dev server caches the OLD Prisma client after a migration (restart; if that fails, clear `.next`)
- **Symptom A (500):** After `prisma migrate deploy` + `prisma generate`, the already-running dev server returns HTTP 500 with `PrismaClientKnownRequestError: The column alumni.<droppedCol> does not exist in the current database` ŌĆö even from a plain `include: { alumni: true }` that names no columns.
- **Symptom B (404):** After `prisma generate` (regenerating `app/generated/prisma/`), the restarted dev server serves the **built-in 404** for *some* route segments only (e.g. `/login`, `/graduates/*`, `/api/alumni-profile`) while others behave normally (`/management/*` ŌåÆ 307 auth-redirect, `/news/*`). Restarting again does NOT fix it. The code is fine ŌĆö `npm run build` + `npm run test` pass and the suspect modules load cleanly in a fresh `node --import tsx` probe ŌĆö so don't chase a code bug.
- **Root cause (A):** `lib/prisma.ts` caches the client in `globalThis.__prisma` (survives HMR). The running process loaded the PRE-migration client into memory; `prisma generate` updates the files under `app/generated/prisma/` but the in-memory singleton still issues SQL for the old columns (e.g. the dropped `maidenLastName`), so every query touching the changed model 500s.
- **Root cause (B):** Turbopack's **persistent** dev cache (`.next/dev/cache/turbopack`) holds a module graph that still references the PRE-regeneration Prisma client files; a plain restart reuses it, so routes whose graph transitively imports the client fail to resolve and Next renders the built-in not-found. Routes that redirect earlier (layout/proxy auth guards ŌåÆ 307) never reach the broken graph, which is why the outage looks selective.
- **Prevention:** After any `prisma migrate`/`generate` against a schema change: (1) restart the dev server (Ctrl+C `npm run dev`, re-run). (2) If you still see 500s *or* selective dev-only 404s, the cache is deeper-stale ŌĆö **`rm -rf .next` then restart** (a plain restart does NOT clear Turbopack's persistent cache). Always verify the data layer independently with a `node --env-file=.env --import tsx` script (uses the fresh client) and confirm `npm run build` + `npm run test` pass before blaming the code. The fresh client (build/test/tsx) is correct even while the dev server is stale. **Docker dev container (the usual setup here):** `app/generated/prisma` is an **anonymous volume** that shadows the host copy, and the running `next dev` process holds the OLD client in `globalThis.__prisma` even after the files regenerate ŌĆö so a host-side `npx prisma generate` leaves the container stale (symptom: `Unknown argument \`<newField>\`` 500s). You must (a) also run `prisma generate` **inside** the container (`docker compose exec app npx prisma generate`) so its volume gets the new client, AND (b) `docker compose restart app` so the process reloads it. Generate on the host too (for `npm run lint`/`build`/`test`). For an additive nullable column the safe non-interactive flow is `migrate status` (drift check) ŌåÆ `migrate dev --create-only` ŌåÆ `migrate deploy` ŌåÆ `generate`; never accept a `migrate reset` on a shared DB.

### News editor toolbar: px font-size ŌĆö selection-only, immediate, and the focus-juggling traps
- **Symptoms (three, fixed together):** (1) the px font-size control did nothing; (2) when it did fire, it restyled the **whole body** instead of just the selected text; (3) attempts to apply "immediately" (on every keystroke) silently set everything to 8 px.
- **Root causes:**
  - **Focus-stealing input discards the selection.** Toolbar buttons (bold/italic/align) keep the contentEditable selection alive via `onMouseDown preventDefault`. The px control was an `<input type="number">`, which **can't** preventDefault (it needs focus to type), so focusing it discarded the editor's selection ŌåÆ `execCommand("fontSize")` ran on an empty selection and produced nothing. `editor.focus()` does **not** bring it back. Fix: cache the last in-editor, non-collapsed `Range` on `onMouseUp`/`onKeyUp` (`captureSelection` ŌåÆ `savedRangeRef`) and restore it before `execCommand`.

Fix: one apply path + a `lastAppliedPxRef` guard so a commit never re-applies the same size.
  - **Never collapse-to-end as a "no selection" fallback.** The old `else` branch did `selectNodeContents(editor); collapse(false)` then `execCommand` ŌĆö on a stale range that restyled everything. `applyFontSize` must **no-op when nothing is selected** (return early); it styles the CURRENT SELECTION ONLY.
  - **onChange-apply is a trap (the 8 px bug).** Applying on the input's `onChange` calls `editor.focus()` inside `applyFontSize`, which **blurs the input ŌåÆ fires `onBlur` synchronously ŌåÆ reads stale state ŌåÆ applies the clamped min (8 px)**, overwriting the real value. execCommand fundamentally needs the editor focused, so you cannot apply-on-keystroke without focus juggling.
- **Prevention (the design that works):** immediate apply is done by dedicated **ŌłÆ/+ buttons with `onMouseDown preventDefault`** ŌĆö the editor never loses focus/selection, so `applyFontSize` runs on the **live** selection with zero focus juggling. The number `<input>` is for exact entry and commits on **Enter/blur only** (`onChange` just tracks the typed text; `onBlur` clamps + guarded-apply). `applyFontSize` prefers a live in-editor selection, else restores `savedRangeRef`, else no-op. **After `execCommand`, the prior selection is collapsed/invalid**, so re-select the just-created `<span>` (`setStartBefore`/`setEndAfter`) and store THAT range ŌĆö repeated ŌłÆ/+ clicks then keep targeting the same text (empirically updates the one span, no nesting). The save/render path is fine: inline `style="font-size"` survives `sanitize-html` at render (v2.17.4 keeps it once `style` is in `allowedAttributes["*"]`) and the news API stores `body` raw.
- **News status default & create options:** `News.status` defaults to `DRAFT`. The form select shows `DISCONTINUED` only when `watch("status") === "DISCONTINUED"` (editing an already-discontinued item, so the field isn't blank). The **list filter** `<select>` keeps all four options (incl. DISCONTINUED) ŌĆö that's a different control.

### Superadmin column-resize ŌĆö overlay-measured grips + `<colgroup>` widths; 7 hand-rolled tables opt in per-page
- **What:** Superadmins can drag a column border (Excel-style) to resize columns on the 7 management data tables (awards, associations, graduate-committee, potentials, model-representatives, all-alumni, alumni-agency). Widths persist **per superadmin in the DB** (`table_column_widths`, JSON `{colIndex: px}` keyed by `userId`+`tableKey`). Non-superadmins see no grips and all-auto columns.
- **Why an overlay, not per-`<th>` handles:** every management table **hand-rolls** its own `<table>`/`<thead>`/`<th>` (the shared `components/data-table.tsx` is unused dead code) ŌĆö there is no column array or single chokepoint. Rather than inject a grip into ~80 individual `<th>`s, a single `<ColumnResizeOverlay>` (`components/table-resize/`) measures the rendered `thead th`s (`getBoundingClientRect`) and floats a drag grip over each header's right border. `<ColGroup>` applies saved widths via `<col>` (extra `<col>` beyond the real column count are harmless). The overlay self-gates on superadmin and re-measures on scroll/resize/width-change, so grips track the dividers without per-column bookkeeping.
- **Resize REQUIRES `table-layout: fixed` (the original "keep auto layout" plan was wrong):** under the default `auto` layout a `<col>` width is only a *minimum hint*, and since every cell is `whitespace-nowrap` each column's min-content width is its full content ŌĆö so dragging does nothing (content holds the column open) and "it's grabbable but can't be dragged." `fixed` makes `<col>` widths authoritative. The hook flips the table to `fixed` **lazily**: the table stays `auto` (identical look) until widths are in play, then `captureDefaults()` snapshots each `<th>`'s current auto width as the per-column default and the hook sets `tableRef.current.style.tableLayout = "fixed"` (imperatively ŌĆö the `<table>` has no `style` prop so React won't clobber it; non-superadmin/un-resized ŌćÆ `""` ŌćÆ auto). `ColGroup` renders `saved[i] ?? default[i]`, so un-resized columns keep their original width and only dragged ones change. Capture happens on first resize, and on mount when saved widths already exist (so a returning superadmin's resizes actually apply). `resetAll` clears both saved AND defaults ŌćÆ back to auto. Pass `tableRef` into `useResizableColumns(key, count, tableRef)` (3rd arg, required) or none of this works.
- **Wiring pattern (per table ŌĆö all 7 are identical):** (1) `const tableRef = useRef<HTMLTableElement>(null); const resize = useResizableColumns("<tableKey>", <thCount>);` (2) add `relative` to the table's `overflow-x-auto` wrapper; (3) on `<table>` add `ref={tableRef}` and `<ColGroup count={N} widths={resize.widths} />` as its first child; (4) render `<ColumnResizeOverlay tableRef={resize={resize}} />` right after `</table>` inside that wrapper; (5) a superadmin-only `ÓĖŻÓĖĄÓ╣ĆÓĖŗÓ╣ćÓĖĢÓĖäÓĖ¦ÓĖ▓ÓĖĪÓĖüÓĖ¦Ó╣ēÓĖ▓ÓĖć` button (`resize.resetAll`) in the action row. The hook (`lib/use-resizable-columns.ts`) keeps widths in the react-query cache (`setQueryData` during drag, debounced `PUT /api/table-prefs` on drag end); `useRole()==="superadmin"` gates; double-click a grip auto-fits that column.
- **Mode-suffixed keys:** `tableKey` must differ when the column set changes ŌĆö `all-alumni:${dedupeView ? "dedupe" : "all"}` and `alumni-agency:${mode}` (mode is `"thailand"|"abroad"`). The `?table=` allowlist in `app/api/table-prefs/route.ts` lists all 9 keys; add a key there when introducing a new table/mode. **When you add/remove a column on one of these tables**, bump the `<ColGroup count>` (over-count is safe; under-count leaves trailing cols auto) ŌĆö the overlay needs no change (it measures live).
- **API:** `/api/table-prefs` is superadmin-only (`checkSuperAdminPermission`), `GET ?table=` + `PUT {table, widths}`; empty widths ŌćÆ delete the row (all-auto). **No activity logging** ŌĆö it's a high-frequency UI pref, not an auditable data mutation.
- **Trap ŌĆö seed the drag with the REAL `<th>` width, never the grip's:** the overlay grips are overlay elements (siblings of the `<table>`, NOT children of the `<th>`), so measuring `e.currentTarget.parentElement` inside a grip pointerdown returns the grip's own ~10px wrapper, not the column. Seed the drag by looking the `<th>` up by index (`tableRef.querySelectorAll("thead th")[i].getBoundingClientRect().width`) ŌĆö that's why the hook's `beginResize(i, startWidth, clientX)` takes the seed width as an argument instead of measuring it. Also keep the per-`<th>` ResizeObserver + `<thead>` MutationObserver: grip positions (`th.right ŌłÆ container.left`) go **stale** after a data load reflows the auto-layout columns, so without re-measuring on `<th>` width changes the grips drift off the real borders and "some columns can't be dragged".
