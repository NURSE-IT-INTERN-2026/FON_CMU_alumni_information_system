## Architecture

### Core Stack

- **Next.js 16.2** with App Router (`app/` directory). This version has breaking changes from earlier Next.js — read the relevant guide in `node_modules/next/dist/docs/` before writing code.
- **React 19.2** with server and client components.
- **Prisma 7** ORM with PostgreSQL. Schema at `prisma/schema.prisma`. Client generates to `app/generated/prisma/` (import from there, not `@prisma/client`).
- **Prisma PostgreSQL adapter** (`@prisma/adapter-pg`) — the Prisma client is initialized with the `PrismaPg` adapter in `lib/prisma.ts`, using `DATABASE_URL` from `.env`.
- **Tailwind CSS 4** with PostCSS plugin (`@tailwindcss/postcss`). CSS-first configuration in `app/globals.css` — no `tailwind.config.js`. Uses `@theme inline` block for custom theme colors (primary: `#5b21b6` purple, accent: `#e8a838` gold).
- **Vitest 4** for testing. Config at `vitest.config.ts`. Tests in `tests/`. Default env is `node` (no DOM); for a DOM-touching test, add `happy-dom` (devDep) and put `// @vitest-environment happy-dom` as the file's first line (see `tests/drag-scroll.test.tsx`).
- **Path alias:** `@/*` maps to project root.

### Prisma Client Import Pattern

```ts
import prisma from "@/lib/prisma";
```

The singleton pattern in `lib/prisma.ts` prevents multiple client instances during hot reload. The client is imported from `../app/generated/prisma/client`, **not** `@prisma/client`.

### Auth & Roles

- **Session-based auth** using `bcryptjs` + HTTP-only cookies (`fon-cmu-session`). Session tokens stored in `Session` model, 7-day expiry.
- **CMU OAuth2** with PKCE via Microsoft Entra ID (`lib/oauth.ts`). Callback at `/api/auth/callback/` (route `app/api/auth/callback/route.ts`; the registered redirect URI is the `CALLBACK_URL` env var, which both the authorize request and the token exchange read).
- **2 Roles:** `superadmin` (full CRUD + user management), `admin` (CRUD + import/export).
- **Alumni portal** (`/graduates/*`): separate self-service session flow — email/password login, **email-verification + admin-approval sign-up** (a signup creates an `UNVERIFIED` account and emails a verification link; clicking it flips to `PENDING`; an admin must then approve on the manage-accounts table before the alumni can log in), first-login TOS acceptance. Reuses the `Session` model with `sessionType: ALUMNI`; auth routes under `app/api/alumni-auth/`.
- **Role context:** `lib/role-context.tsx` provides `useRole()`, `useCanWrite()`, `useIsAdmin()` hooks.
- **Write permission check:** `lib/permissions.ts` — `checkWritePermission()` returns 401/403 for unauthorized requests.
- **Rate limiting:** `lib/rate-limit.ts` — in-memory sliding-window (5 attempts / 15 min).
- **Auth middleware via `proxy.ts`** (Next.js 16 renamed middleware to proxy) — enforces CSP headers and redirects unauthenticated users. Also enforced at the layout level: `app/(admin)/layout.tsx` guards the admin area; `app/graduates/(authed)/layout.tsx` guards the alumni portal.

### API Route Pattern

Each data entity follows a consistent route structure:
- `GET /api/{entity}` — list (paginated, searchable)
- `POST /api/{entity}` — create
- `GET/PUT/DELETE /api/{entity}/[id]` — read/update/delete
- `POST /api/{entity}/import` — Excel import
- `GET /api/{entity}/export` — Excel export
- `POST /api/{entity}/bulk-delete` — bulk delete by IDs
- `POST /api/alumni/create-with-related` + `PUT /api/alumni/update-with-related/[id]` — create/update an alumni together with related records (full-form; one save can affect other pages)
- DELETE is a **soft delete**; recovery is superadmin-only via `POST /api/trash/restore` and `POST /api/trash/hard-delete`
- Every mutating route must call `checkWritePermission` (`@/lib/permissions`) and `logActivity` (`@/lib/activity-log`)
- **Standard CRUD + import/export/bulk-delete entities:** alumni, alumni-agency, associations, awards, graduate-committee, model-representatives, potentials.
- **Deviations from the standard pattern:**
  - `alumni-activity` — read-only engagement analytics (login activity + accounts); no `/[id]`/import/export/bulk-delete. Session-guarded + 60s-TTL-cached (`withTtlCache("alumni-activity")`), same class as `dashboard`/`alumni-count`. Login counts read `ActivityLog` where `action='LOGIN'` + `resource='alumni_auth'` + `actorType='ALUMNI'` (see Known Pitfalls).
  - `news` — no `import`/`export`; DELETE → `status: DISCONTINUED` (NOT a soft delete, NOT trash-recoverable). **Pinned news ("ประชาสัมพันธ์สำคัญ"):** `GET /api/news` **excludes pinned by default** (`pinnedAt: null`) and `?pinned=true` returns only pinned (`pinnedAt desc`, used by the top-of-page section on both the admin + alumni news pages); `POST /api/news/[id]/pin` `{ pinned }` is the admin toggle. Discontinuing (single DELETE / bulk-delete / PUT status→DISCONTINUED) clears `pinnedAt` (pinned ⇒ not-discontinued).
  - `users` — no `import`/`export`/`bulk-delete`; write ops are superadmin-only (`checkSuperAdminPermission`).
  - `alumni-accounts` — admin alumni-account mgmt; no import/export/bulk-delete. The list filters `passwordHash != null` **and excludes `UNVERIFIED`** (`accountStatus: { not: "UNVERIFIED" }`) — admins track only accounts that cleared email verification (PENDING/ACTIVE/REJECTED); `?status=` accepts `pending|active|rejected` (NOT `unverified`), and `UNVERIFIED` is surfaced in no admin-tracking surface (this list / the dashboard count / the alumni-activity count). Returns `accountStatus` + `signupVerification` + `rejectionReason` + `rejectedAt`. Sub-routes: `/[id]/suspend` (toggle `suspendedAt` + kill sessions), and the **admin-approval** set `/[id]/approve` (PENDING|REJECTED → ACTIVE; creates the deferred Education + `syncPrimarySnapshot` + approval email; clears `rejectionReason`/`rejectedAt`), `/[id]/reject` (**requires `{reason}`** — zod non-empty; → REJECTED + kill sessions + stamps `rejectionReason`/`rejectedAt` + rejection email w/ the reason + a ยื่นคำขอใหม่ link; re-approvable; the review modal shows the latest reason + the REJECT history from `GET /api/alumni/[id]/activity`), `/[id]/reverify` (re-fetch CMU, rebuild `signupVerification`). All write sub-routes use `checkWritePermission` (admin + superadmin) + `logActivity` (`APPROVE`/`REJECT`/`SUSPEND`/`RESTORE`). **Exception — `/[id]/delete`** is the one **superadmin-only** sub-route (`checkSuperAdminPermission`): it deletes the alumni *account* (login credentials) while **KEEPING the data record** — nulls `email`+`passwordHash`+`accountStatus`→`UNVERIFIED`+`signupVerification`+`emailVerifiedAt`+`suspendedAt`+`rejectionReason`+`rejectedAt`+`hasLoggedIn`+`lastLoginAt`+`tosAcceptedAt`, does NOT set `deletedAt` and does NOT touch `studentId`/name/education; in one `$transaction` it also `deleteMany`s the alumni's `Session`s + `EmailVerification` + `PasswordReset` tokens, then `bustCache("dashboard")`+`bustCache("alumni-activity")` + logs `DELETE`/`alumni_auth` `{ source: "admin_delete_account" }` (no email sent). Purpose: let the alumni re-sign-up to re-test the signup email — because signup dedups on `email` first (`findFirst`+`@unique`), nulling `email`+`passwordHash` makes the re-signup take the UPDATE branch on the preserved `studentId` row and re-fire `sendEmailVerificationEmail`. The UI is a superadmin-only (`useRole()==="superadmin"`) trash icon in `AlumniAccountsTab` (`settings/users/page.tsx`) with a confirm modal explaining the record is kept + re-sign-up enabled.
  - `alumni-profile` — no `/[id]`; operates on the logged-in alumni (`getAlumniSession`). GET/PUT/DELETE. `alumni-profile/educations` adds alumni-self GET/POST of education records.
  - `cmu-alumni` — read-only (GET list/search) + `lookup?studentId=&alumniId=` (single-record preview for the add-education form; with `alumniId`, returns `samePersonWarning` if the record belongs to a different person, and `alreadyClaimed` if the studentId is already another alumni's education record) + `/live` (session-gated LIVE CMU list, server-paginated/searchable, `withTtlCache("cmu-live-graduates", 120s)`-wrapped so one CMU call serves all pagination/search within the window; each row carries `isNew` = its trimmed `student_id` is NOT in the local `cmu_graduates` cache — powers the cmu-sync "ข้อมูลล่าสุดจากทะเบียน" table). `fetchCmuGraduatesLive` has no cache of its own, so the live table MUST go through this cached route, not call it directly. The cmu-sync "ข้อมูลในระบบ" table just hits the local `GET /api/cmu-alumni?dedupe=false` (raw records → total = `localCount`).
  - `educations` — degree records (1:N per alumni). `GET/POST /api/alumni/[id]/educations` (admin) and `GET/POST /api/alumni-profile/educations` (alumni-self) for list/add; `GET/PUT/DELETE /api/educations/[id]` for one record (admin OR owning alumni via `resolveWriter`). Every add, and a PUT that changes `studentId`, must pass `assertEducationSamePerson` (`lib/education-identity.ts`) — a studentId whose CMU birthday differs from the alumni's is rejected with 400 (can't attach a stranger's degree). No import/export/bulk-delete.
  - `logs` — otherwise read-only (GET list only). Adds a **superadmin-only** `POST /api/logs/bulk-delete` `{ ids }` that **hard-deletes** (`deleteMany`; `ActivityLog` has no `deletedAt`). Not soft-delete, not trash-recoverable, and **deliberately does NOT log the deletion** (logging it would defeat the purpose of removing log entries). UI gate on the logs page is `useRole() === "superadmin"` (NOT `useIsAdmin()`, which covers both roles).

### Route Correlations, Redirects & Entry Points

When adding/changing a route, keep this map honest (see Working Protocol "On touching routes").

**Correlations — routes that pair or overlap (touching one affects the other):**
- **Soft-delete round-trip:** `DELETE /api/{entity}/[id]` + `/api/{entity}/bulk-delete` set `deletedAt`; recoverable only via `POST /api/trash/restore` (+ permanent `POST /api/trash/hard-delete`, superadmin). **Exception:** `news` DELETE → `DISCONTINUED`, never trash-recoverable.
- **Full-form vs single CRUD:** `POST /api/alumni/create-with-related` + `PUT /api/alumni/update-with-related/[id]` write Alumni AND its 6 related entities in one transaction — overlaps `/api/alumni` POST/PUT and the per-entity routes; one save can change several pages.
- **Admin alumni profile view:** `/management/alumni/[id]` (param = alumni UUID **or** `studentId` — `GET /api/alumni/[id]` resolves either) is reached by clicking any row on the all-alumni + alumni-related tables; its data-logs tab reads `GET /api/alumni/[id]/activity` (field-change history ∪ activity log). Editing there saves via `PUT /api/alumni/update-with-related/[id]` (overlaps the full-form route; sets `adminEditedAt`). Replaces the deleted `/management/settings/alumni/[id]`. Orange indicators on this page query BOTH `resourceType: alumni` and `alumni_profile` (admin + alumni-self edits) — `OrangeCell`/`FieldHistoryModal`/`/api/field-changes` accept comma-joined `resourceType`.
- **Alumni import columns:** `POST /api/alumni/import` reads `รหัสนักศึกษา/คำนำหน้า/ชื่อ/นามสกุล/รุ่น สาขา/ระดับการศึกษา/อีเมล/เบอร์โทร/ที่อยู่ปัจจุบัน`. `อีเมล` is the CONTACT email → written to `Alumni.contactEmail` (NOT the auth `email`, which is the login identity, `@unique`). `เบอร์โทร` may hold several numbers (comma-separated, possibly with a `มือถือ` label) → `parsePhones` (`lib/parse-phone.ts`) keeps only the mobile after `มือถือ` and splits commas into `Alumni.phones` (`String[]` — never a clumped string). `ที่อยู่ปัจจุบัน` → `homeAddress`. `scripts/build-alumni-excel.ts` emits these from the legacy `Tbl_temp` dump, cleaning the dump's multi-dialect encoding corruption along the way: digit-mojibake (`İ–Ĺ`→`0–9`, on `StudID`/`TypeEdu` only), **Malayalam-block Thai corruption** (Thai chars that landed at U+0D00–0D7F are shifted back +0x100 to valid Thai — e.g. `ปรีชาพงค์มิตണ→ร`), and **unrecoverable row-concatenation binary garbage** (CJK/Tibetan/Latin-1 clusters, `℡`/`⌀` separators — dropped from text fields, not a reversible transform). Re-run `node --import tsx scripts/build-alumni-excel.ts` to regenerate a clean `imports/excels/alumni.xlsx` (the file is gitignored — the script is the source of truth).
- **Account lifecycle:** `POST /api/alumni-accounts/[id]/suspend` toggles suspension AND kills active `Session`s; `PUT /api/users/[id]` does the same for admin users. **Signup approval (two gates — email verification THEN admin approval):** `/api/alumni-auth/signup` creates an `UNVERIFIED` account + an `EmailVerification` token + sends a verification email (no auto-login, no session, Education deferred) → alumni clicks the link → `POST /api/alumni-auth/verify-email` flips it to `PENDING` (`emailVerifiedAt` set; enters the admin queue) → `/[id]/approve` (ACTIVE) ↔ `/[id]/reject` (REJECTED — **requires a reason**; re-approvable) → `POST /api/alumni-auth/reapply` (alumni proves identity with email+password; **no re-verification** — email was already verified) flips REJECTED → PENDING so a corrected application re-enters the queue. Only `accountStatus === ACTIVE` may log in — `login-email` returns 403 for UNVERIFIED (with `code:"UNVERIFIED"`, so the login page offers a resend), PENDING (no code), and REJECTED (with `code:"REJECTED"`, so the login page + rejection email offer ยื่นคำขอใหม่ → `/graduates/reapply`), and `getAlumniSession` also rejects non-ACTIVE. `POST /api/alumni-auth/verify-email/resend` re-issues a token (enumeration-safe; only for UNVERIFIED). Admins may also `approve` straight from UNVERIFIED (override; sets `emailVerifiedAt`). Education/snapshot creation happens at **approve**, not signup, so a rejected signup never pollutes degree data. **Password reset:** `POST /api/alumni-auth/forgot-password` issues a 1-hour token to ACTIVE **or REJECTED**, non-suspended alumni with a `passwordHash` (enumeration-safe — same message regardless of why it doesn't send; REJECTED is admitted so a rejected user who forgot their password can reset it and return to reapply — resetting does NOT change accountStatus), **invalidates prior unused tokens**, and sends the email best-effort (a send failure is logged, not thrown). `POST /api/alumni-auth/reset-password` validates the token, re-checks ACTIVE-or-REJECTED/non-suspended (defense in depth — mirrors the forgot step; the account's status may change between link issue and click), requires `password === confirmPassword` server-side, hashes the new password, marks the token `used`, and deletes all the alumni's `Session`s (force re-login). Both are rate-limited per IP and logged (`PASSWORD_RESET_REQUEST` / `PASSWORD_RESET_COMPLETE`).
- **CMU lookup:** `/api/cmu-alumni` (list/search) + `lib/cmu-registrar.ts` (`fetchCmuGraduateById`) feed import major-sync (`lib/ensure-alumni.ts`), signup verification (`/api/alumni-auth/signup`), and the add-education auto-fill preview (`/api/cmu-alumni/lookup`, which reuses `cmuToAlumniFields` and, given `?alumniId=`, returns `samePersonWarning` + `alreadyClaimed` so the form can warn before the POST guard rejects).
- **Education / primary snapshot:** the primary is the **highest** degree and is auto-derived — `recomputePrimaryEducation` (`lib/education-sync.ts`) runs after every add/edit/delete (PUT/DELETE `/api/educations/[id]` inside the route's `$transaction`; the add routes + signup `approve` after the create), sets `primaryEducationId` to the highest-degree row, then `syncPrimarySnapshot` mirrors it onto the denormalized `Alumni` snapshot. That snapshot's `studentId` is the FK target for the 6 related tables (Award/Association/…) + what the all-alumni table reads — so promoting a higher degree to primary cascades exactly like editing `Alumni.studentId` directly (related rows follow via `ON UPDATE CASCADE`). It can't be set to a lower degree by hand (no field/endpoint exposes `primaryEducationId`). Shared UI: `components/EducationSection.tsx` (add/edit dialogs, badges the primary), wired into both `/management/alumni/[id]` and `/graduates/(authed)/profile`.
- **Twin auth:** `/api/auth/*` (ADMIN) vs `/api/alumni-auth/*` (ALUMNI) share the `Session` model, split by `sessionType`.

## Known Pitfalls & Lessons Learned

*(Append-only ledger of project-specific traps. See "Self-Improvement Loop" for when to add an entry.)*

### `docker compose up` reuses a stale cached app image — rebuild with `--build` or the app serves old code
- **Symptom:** Docker says the app is "running on :3000" and a Next.js server responds, but the app is broken — it serves at the root (`/login` returns 200) instead of under `basePath: "/alumni"` (`/alumni/login`), so every internal link/fetch (all written for `/alumni`) 404s/redirects wrong and the page won't load. Confirmed via curl: `/` → 307 `/login` (not the expected 308 → `/alumni`), and bare `/login` → 200.
- **Root cause:** `basePath` is a **build-time** constant baked into the standalone image (`output: "standalone"`). `docker compose up -d` reuses an existing image of the same name — it does NOT rebuild from source. After pulling/merging code changes, an image built weeks earlier keeps running: `docker images` shows `fon-cmu-alumni-app:latest` created "4 weeks ago" while the container is minutes old. The host dev server (`npm run dev`) is NOT involved — `ps` shows no `next`/`node` process; the `*:3000` listener is the container's published port owned by the Docker Desktop VM.
- **Prevention:** After any source change you want live in Docker, run `docker compose up -d --build app` (or `docker compose build app` then `up -d app`). Confirm the image is fresh (`docker images` → created "x minutes ago", new image id). Verify basePath with curl: `/` → 308 `/alumni`, `/alumni/login` → 200, bare `/login` → 404. `next.config.ts`'s `basePath` is NOT runtime-overridable, so a wrong basePath always means a stale build, never an env tweak.

### Dev container hot-reloads via `docker-compose.override.yml` (HMR) — but only after a `--build`
- **What:** `docker compose up` (no `-f`) auto-merges `docker-compose.override.yml`, which overrides the `app` service to build the Dockerfile **`deps`** stage (full deps, incl. devDependencies) and run `next dev` with the source **bind-mounted** (`.:/app`) — so code changes hot-reload like a normal host `npm run dev`. Anonymous volumes protect `/app/node_modules`, `/app/.next`, and `/app/app/generated` from the bind mount (container-owned). The command runs `npx prisma generate` (the `deps` image has no generated client) then `npm run dev -- -H 0.0.0.0`.
- **The catch:** the stale-image rule ABOVE still applies on the **first** switch to dev (or after any `package.json` change) — `docker compose up` alone reuses the old standalone image, so you MUST run `docker compose up -d --build app` once. After that, plain `docker compose up` reuses the dev image and source edits reload via HMR with no rebuild. To go back to the production standalone container, deploy with `docker compose -f docker-compose.yml up -d --build app` (the `-f` skips the override).
- **Verify HMR:** the dev server logs `✓ Ready in …ms`; editing a host source file then requesting its route logs `○ Compiling /<route> …` (on-demand recompile). Confirm the container reads live source, not a baked copy: `docker compose exec app grep -c '<recentToken>' '/app/app/(admin)/management/alumni-agency/page.tsx'` (>0).
- **New npm dependency (gotcha):** a host-side `npm install` / `npx shadcn@latest add …` updates `package.json` + `package-lock.json` (bind-mounted, so the container sees them) AND the *host's* `node_modules` — but NOT the container's `node_modules` (anonymous volume). The dev server then **500s** with `Module not found: Can't resolve '<pkg>'`. A host `npm run build`/`npm run lint` still **passes** (it uses host `node_modules`) — misleading, since the running dev server is the container. Fix: `docker compose exec app npm install` to install into the container's volume, then the route recovers. (Deps that resolve via the `radix-ui` umbrella need no extra direct dep — only genuinely-new packages trigger this.)

### Dev container `public/uploads/` is a named volume — host files there are NOT served
- **Symptom:** An image that clearly exists at `public/uploads/x.png` on the host 404s when fetched from the running dev container (`GET /alumni/uploads/x.png → 404`), while app-uploaded images serve fine. Looks like the upload/thumbnail feature is broken when it isn't.
- **Root cause:** `docker-compose.yml` mounts the named volume `uploads` at `/app/public/uploads` (to persist uploads in production). The dev override bind-mounts `.:/app` for HMR but does NOT remove that named volume, and compose **merges** (not replaces) volumes — so the named volume **shadows** the host's `public/uploads/` inside the container. The container's `/app/public/uploads/` is the volume's own (root-owned) tree, separate from the host directory. App uploads write to `process.cwd()/public/uploads` = the volume, so they serve and persist; host-side files are invisible to the container.
- **Prevention:** When verifying uploads/thumbnails against the dev container, don't drop a file into the host `public/uploads/` and expect the container to serve it. Either upload through the app (`POST /api/upload`), or place the file inside the container: `docker cp file.png fon-cmu-alumni-app-1:/app/public/uploads/file.png` (and `docker exec … rm` it after). To see what the container actually has: `docker exec fon-cmu-alumni-app-1 ls /app/public/uploads/`.

### App is deployed under `basePath: "/alumni"`
- **Symptom:** Links, redirects, or fetches resolve to the wrong path in the deployed app.
- **Root cause:** `next.config.ts` sets `basePath: "/alumni"`. Route files define paths relative to it; some API helpers auto-prepend `BASE_PATH`, others need it manually.
- **Prevention:** Account for `basePath` in new links/calls. Don't hardcode absolute `/api/...` blindly — check whether the helper already prepends.

### Shared `lib/` files must stay client-safe (split Prisma into a `-server` module)
- **Symptom:** Build error `Can't resolve 'dns'` (or another Node-only module) traced to a client component.
- **Root cause:** A client component imported a `lib/` module that transitively pulls in Prisma / the `pg` driver.
- **Prevention:** Keep shared `lib/` modules client-safe. Put Prisma-backed logic in a dedicated `*-server.ts` file (pattern in use: `lib/filter-facets.ts` + `lib/filter-facets-server.ts`). Never import `@/lib/prisma` across a `"use client"` boundary.

### API routes swallow Prisma errors into generic 500s
- **Symptom:** A data API returns a blank 500 with no useful detail.
- **Root cause:** Route handlers catch Prisma errors and return a generic message, hiding the real cause.
- **Prevention:** When debugging an API failure, probe the Prisma layer directly (run a script with `node --env-file=.env --import tsx`) rather than trusting the API response. The runtime DB is remote Prisma Postgres.

### Schema edits require a created + applied migration on remote Prisma Postgres
- **Symptom:** A model/schema change doesn't take effect, or the generated client disagrees with the live DB.
- **Root cause:** `npx prisma generate` alone is not enough — the migration must be created AND applied to the remote DB, then the client regenerated.
- **Prevention:** After any `prisma/schema.prisma` change: create the migration, apply it to the remote DB, then `npx prisma generate`. Run DB scripts with `node --env-file=.env --import tsx`.

### model-representatives เครือข่าย comes from alumniNetwork.aspx, not CMU degree level
- **Symptom:** model-representatives `เครือข่าย` (stored in `ModelRepresentative.cohort`) showed fabricated values (all `ปริญญาโท`).
- **Root cause:** `imports/finalized/model-representatives.xlsx` held 15 fabricated test rows from `scripts/build-test-imports.ts` → `cohortLabel()`, which derives the network from CMU degree `level_id`. The real per-network reps live in `imports/scrapped/alumni-network.json` (scraped from `alumniNetwork.aspx`, section titles already stripped of `รายชื่อเครือข่ายศิษย์เก่า` + `(รุ่น …)`). `imports/` Excel files are test/fabricated artifacts — not the source of truth.
- **Prevention:** `เครือข่าย` is one of 5 networks only (ปริญญาพยาบาล / ผู้ช่วยพยาบาล / อนุปริญญาพยาบาล / ปริญญาโท / ปริญญาเอก) — never derived from degree level. Rebuild from the real source via `node --env-file=.env --import tsx scripts/rebuild-model-representatives.ts` (name-joins reps → studentId via `alumni-data.json`; reports unmatched to `alumni-network-unmatched.json`). `DRY_RUN=1` previews counts without touching the DB.

### model-representatives `cohort`=เครือข่าย, `generation`=รุ่นที่ (inverted vs. sibling entities)
- **Symptom:** FacetFilters showed wrong data — เครือข่าย filter listed names, รุ่นที่ filter listed เครือข่าย values.
- **Root cause:** PRD names the fields `network` (เครือข่าย) + `cohort` (รุ่นที่), but the `ModelRepresentative` schema stores เครือข่าย in **`cohort`** and รุ่นที่ in **`generation`** — the opposite of sibling entities (graduate-committee etc.), where `cohort` = รุ่นที่. Filters/columns copied from siblings land on the wrong field.

- **What's split:** `Award` (was `recipientName`), `Potential`/`Association`/`GraduateCommittee` (were `fullName`), `ModelRepresentative` (was `name`), `AlumniAgency` (was `thaiName`; keeps `englishName` + its own `prefix`). A "เลือกทั้งหมดในหน้านี้" header button (select mode only) replaces the old select-all checkbox — it toggles `selectAll(pageIds)`/`deselectAll()` via `isAllSelected`. Per-page notes: alumni-agency's BOTH tabs (Thailand + abroad) share one CRUD surface with selection (see the in-country lesson below); news is card-based (select-all lives in the header, not a card-area checkbox). Caveat: the orange selection highlight is the same color family as OrangeCell's "recently edited" cell indicators — both are orange.
- **What:** The all-alumni page (`app/(admin)/management/all-alumni/page.tsx`) renders ONE table with inline edit + row-click → `/management/alumni/[id]` + create → `/management/new-alumni`. `manageQuery` fetches the FULL CMU list once (query keyed **without `page`** — only `search`/`filters`/`sort`; `pageSize=50000` since `/api/cmu-alumni` caps at `MAX_PAGE_SIZE=50000`) AND the full local list (`/api/alumni?pageSize=50000&includeDeleted=true` — the `50000` is load-bearing: there are >11k local alumni and the old `9999` silently **truncated ~1,600 records**), merges the complete set into **`Alumni` shape**, then slices client-side (`allMerged.slice((page-1)*PAGE_SIZE, page*PAGE_SIZE)`) — so navigating pages is instant (no refetch). **Local-only rows are included** (alumni whose `studentId` is NOT in CMU — admin-created or legacy-Excel-imported; ~4,167 as of 2026-06): a secondary pass appends each one, otherwise created/imported alumni are **invisible** in the table. Facets filter server-side; `allMerged` (the full set, not the page slice) is used for delete/bulk-delete lookups so CMU-only rows selected on other pages still resolve.

Note: `normalizeCmuBirthday` was originally (wrongly) coded as MM-DD-YYYY; the form-side parser `normalizeFormBirthDate` was already day-first, so the MM-DD assumption made signup `birthDatesMatch` inconsistent — DD-MM is correct. **`Alumni.birthDate` (the local DB column) is stored as Buddhist-era DDMMYYYY (8 digits, e.g. `01122540`) — NOT canonical YYYY-MM-DD** (the schemas enforce `/^\d{8}$/` and write paths store it raw); `formatBirthDateThai`/`formatBirthDateThaiSlash` normalize it via `normalizeFormBirthDate` before display (canonical `YYYY-MM-DD` is also accepted), so the profile/table render it correctly. - **Trap:** Adding a sortable column must keep it reachable by the client-side sort (`sortAlumni` on the `Alumni`-shaped merged set). Don't page the CMU proxy one page at a time and merge per-page — local-only rows would repeat across pages and the sort would be wrong. Fetch the **full** set once, merge, sort, then slice client-side.

### CMU returns one record per degree → dedup-by-person is DISPLAY-LAYER ONLY
- **Symptom:** The same person shows up as multiple rows (and inflates counts) because they earned more than one FON degree — e.g. a Bachelor's then a Doctoral. Each degree is a separate CMU record with a **distinct `student_id`** and different `level_id`.
- **Root cause:** `fetchCmuGraduates()` (`lib/cmu-registrar.ts`) returns the raw per-degree list; nothing in the upstream collapses them. Display/count surfaces were reading it verbatim, so one person counted N times.
- **Prevention:** Use `dedupeCmuGraduatesByPerson(graduates)` (`lib/alumni-verify.ts`) — matches on `name_th`+`surname_th`+canonical birthday (NUL-joined), keeps the **highest** degree (`DOCTORAL>MASTER>BACHELOR>ASSOCIATE>NURSING_ASSISTANT`), and leaves incomplete-identity records verbatim. **Apply it ONLY in the table + facet display surfaces**: `/api/cmu-alumni` (on the full list, *before* search/facets/sort/pagination so records on different pages still collapse) and `lib/filter-facets-server.ts` (`getAlumniFacetValues`, so facet counts match the table). The dashboard **count** routes (`/api/alumni-count`, `/api/dashboard`) instead use the merged person count below — NOT this dedup. **NEVER** apply it in studentId-keyed consumers — `lib/ensure-alumni.ts` (the import studentId→record map) and `app/api/alumni/[id]/route.ts` (profile lookup by studentId) MUST keep the raw `fetchCmuGraduates()`, because dedup drops each person's lower-degree `student_id`s and would silently break CMU enrichment on import / profile view. `fetchCmuGraduates()` itself stays un-deduped (its 5-min cache serves both the deduped and the raw consumers).

### CMU `student_id` has surrounding spaces — the list proxy trims; direct consumers trim too
- **Symptom:** The all-alumni table over-counted vs the dashboard (26,068 vs 20,988) and the alumni-search dropdown showed duplicate suggestions for the same person.
- **Root cause:** `fetchCmuGraduates()` returns `student_id` RAW (e.g. `"137828   "` — ~6,295 of 16,821 deduped records carry spaces); local `Alumni.studentId` is clean. The all-alumni table and `useAlumniSearch` joined/merged on the raw CMU id, so the CMU↔local overlay missed and each such alumni became TWO rows (a CMU-only row + a local-only row — ~4,768 duplicates). The dashboard (`groupPersonsByDegree`), `ensure-alumni`, and `fetchCmuGraduateById` already `.trim()` defensively, which is why only the table/search were wrong.

If you consume `fetchCmuGraduates()` DIRECTLY (a lib/server route), you MUST `.trim()` the id before comparing or keying. Never drop the proxy's trim. (The last row-vs-dashboard gap — multi-degree alumni — is closed by the `student_ids` collapse in the next lesson.)

### all-alumni table collapses multi-degree alumni to one row (matches the dashboard)
- **Symptom:** After the local-only + trim fixes the table still over-counted the dashboard by ~312 — multi-degree alumni rendered twice (a CMU row for one degree + a local-only row for another degree of the same person).
- **Root cause:** The table bridged a local alumni to its CMU person on the KEPT (highest) CMU `student_id` only. `dedupeCmuGraduatesByPerson` keeps one record per person and DROPS the lower-degree records (each carries its own `student_id`), so an alumni whose local `education` matched a DROPPED CMU degree couldn't be linked to the kept row → it fell through to the local-only pass → duplicate row. The dashboard's `groupPersonsByDegree` unions all degrees and never had this gap.
- **Prevention:** `dedupeCmuGraduatesByPerson` now attaches ALL of the person's `student_id`s to the kept record as `student_ids` (the `CmuGraduate` type carries the field; the proxy passes it through). `/api/alumni` returns each alumni's `educations` studentIds. The table builds a CMU sid set from every record's `student_ids` and marks an alumni "used" (not local-only) when ANY of its education studentIds is in that set — so each person shows once and the table total equals the dashboard (20,988). `findLocalByCmuRecord` only picks which alumni's data overlays a given CMU row (display). The single `manageQuery` merge carries this collapse logic — keep it in sync with the dashboard's `groupPersonsByDegree`.
- **Opt-out toggle ("แยกรายการตามรายวุฒิ" → now a mode button `แสดงวุฒิสูงสุด` / `แสดงทุกวุฒิ`):** the all-alumni page has a mode button (orange `แสดงวุฒิสูงสุด` ↔ purple `แสดงทุกวุฒิ`) that sets `dedupeView=false` → passes `?dedupe=false` to `GET /api/cmu-alumni` (skips `dedupeCmuGraduatesByPerson`) so every degree record shows as its own row (a multi-degree person appears once per degree). In that mode the merge builds each row's degree fields from the CMU record (`studentId`/`degreeLevel` via `cmuLevelToDegree`/`major`/`graduationYear`/`cohort`) and overlays local identity+contact only; the row `id` stays the local UUID (edit/delete/navigation unchanged) and the React key is `${id}-${studentId}` so a person's degree rows don't collide. The merge **also expands each local alumni into one row per `education`** (degrees whose `studentId` isn't in CMU — a local-only lower degree was otherwise invisible in show-all), and with a search term keeps only the degree row whose `studentId` or the alumni's name matches; `GET /api/alumni` search matches `education.studentId` too (so a non-primary degree id is findable). Facets follow via `?dedupe=false` on `GET /api/filter-facets` (`getAlumniFacetValues` tallies each raw record's `cmuFieldValue`). The **dashboard graph is unaffected** — `/api/alumni-count` + `/api/dashboard` use `getPersonDegreeBreakdown()`, a separate path with no `dedupe` param.

### all-alumni search: filter params must be applied to BOTH the CMU and local fetches
- **Symptom:** Searching on `/management/all-alumni` returned the full unfiltered list — e.g. a local-only `studentId` (a mock `66123456`) yielded every record instead of one.
- **Root cause:** the `manageQuery` queryFn fetches CMU (`/api/cmu-alumni`) and local (`/api/alumni`) separately and merges client-side. `search` was passed to the CMU `URLSearchParams` but **not** the local one, so `/api/alumni` returned all rows; for a local-only query the CMU side was empty and every local alumni survived the merge (none marked "used").
- **Prevention:** when touching the all-alumni `manageQuery` (or any dual-source client-side merge), keep `search`/facet params **symmetric** across every fetch. `/api/alumni` already supports `search` (matches `firstName`/`lastName`/`studentId`), and `search` is already in the react-query key, so the fix was one line — add it to the local params. The other table pages route through `useEntityList` or a single `useQuery`, so they don't have this split-source trap.

### Table search boxes submit on Enter / ค้นหา — never per keystroke

### Dashboard "alumni by degree" counts merge CMU + local (a person never counts twice)
- **Symptom:** The dashboard must count each PERSON once under their highest degree — across CMU AND local `education` rows (a locally-added higher degree upgrades the person; nobody is double-counted when in both sources).
- **Root cause:** CMU (the FON graduate universe) and the local `education` table overlap (signed-up/imported alumni are a subset of CMU). Counting each source separately double-counts the overlap and ignores locally-added degrees.
- **Prevention:** Use `getPersonDegreeBreakdown()` (`lib/person-degree-count.ts`) in `/api/alumni-count` and `/api/dashboard`. Its pure core `groupPersonsByDegree(cmu, local)` unions entities via three signals — CMU records sharing normalized name+birthday, a local alumni's `education` rows (one `alumniId`), and a cross-source **studentId bridge** (a local education whose `studentId` matches a CMU record joins that CMU person). Each group → one person, highest degree (`DEGREE_RANK` from `lib/alumni-verify.ts`), representative year = that degree's most recent year. **The count includes EVERY person** (stays consistent with the all-alumni table, which now shows local-only rows): each person is still flagged `hasCmu` (informational only), and `getPersonDegreeBreakdown` counts all of them — CMU persons AND local-only persons (no CMU record, e.g. a legacy Excel import or an admin-created alumni like `studentId=66123456`). (Historically local-only persons were EXCLUDED — a one-line `if (!p.hasCmu) continue` guard — to avoid inflating the total above the registrar count; that was **reverted 2026-06-30** because it hid ~4,167 real imported alumni from both the table and the dashboard. `hasCmu` is still computed for observability/testing but no longer filters.) The table (`/api/cmu-alumni`) + facets still use `dedupeCmuGraduatesByPerson` (CMU-only). **Note:** alumni-count's `DEGREE_ORDER`/labels are now only used for response shaping, not degree resolution. The CMU fetch inside `getPersonDegreeBreakdown` is **fail-safe** (see the next lesson) — a Registrar outage degrades to local-only counts instead of 500-ing.

### CMU Registrar data is MATERIALIZED locally (`cmu_graduates`) — not fetched live on load
- **What changed (2026-07):** CMU graduates used to be fetched LIVE on every dashboard / all-alumni-table load (virtual, never persisted, 5-min in-memory cache in `lib/cmu-registrar.ts`). They are now **persisted** in a `cmu_graduates` table (one row per FON degree record, `studentId @unique`) and refreshed **on demand** by an admin from the **การดึงข้อมูล** page (`/management/settings/cmu-sync`): it auto-compares local vs remote (studentId-set + count) and a ดึงข้อมูล button materializes the full remote set. The **only** remaining live-CMU call is `fetchCmuGraduatesLive()` inside `POST/GET /api/cmu-alumni/sync`.
- **The rewire trick:** the `CmuGraduate` interface + all PURE consumers (`dedupeCmuGraduatesByPerson`, `groupPersonsByDegree`, `cmuToAlumniFields`, sort/facet logic) are byte-for-byte unchanged. `cmuGraduateRowToShape(row)` maps a persisted row INTO that shape (12 stored fields; ~18 unused API fields are optional on the interface). `fetchCmuGraduateById` / `fetchCmuGraduatesOrEmpty` **keep their names** with bodies switched to local reads — so ~11 of 17 call-sites needed **zero** code change. The 4 consumers that needed an edit: `lib/filter-facets-server.ts`, `app/api/cmu-alumni/route.ts`, `app/api/alumni/[id]/route.ts` (PUT studentId dup check), `lib/ensure-alumni.ts` (dropped `cmuMapPromise`/`resetCmuLookupCache` — the DB is the cache).
- **`cmuAvailable` re-pointed:** previously "Registrar reachable"; now "the `cmu_graduates` table is non-empty". Pre-first-sync → `false` → the dashboard amber banner says "ยังไม่ได้ดึงข้อมูล CMU" (links to `/management/settings/cmu-sync`) and counts local alumni only.
- **Identity checks read local now (full removal):** signup/approve/reverify/education same-person + the alumni `[id]` PUT studentId check all read the local table. A brand-new graduate in remote CMU but not yet synced won't be found — those checks already fail open and are admin-gated. To see new graduates, an admin syncs first.
- **Comparison excludes local-not-CMU data by design:** the diff is remote studentId-set vs `cmu_graduates` studentId-set. Local-only alumni (in `alumni`, no CMU record) are not in `cmu_graduates`, so they never affect in-sync. Removed registrar rows are reported (`removedCount`) but NOT auto-soft-deleted (keeps the table a faithful superset).
- **Sync mechanics:** `GET /api/cmu-alumni/sync` = compare (returns `{remoteCount,localCount,newCount,removedCount,inSync,lastSyncedAt,sample}`); `POST` = chunked 500-row `$transaction` upserts (not `createMany` — must refresh stale fields), `logImport(cmu_alumni/IMPORT)`, `bustCache("dashboard")`+`bustCachePrefix("alumni")`. Auth: admin session (`checkWritePermission`) OR `CMU_SYNC_SECRET` bearer (cron). Dev seed: `node --env-file=.env --import tsx scripts/seed-cmu-graduates.ts` (`DRY_RUN=1` to preview).
- **Supersedes** the "virtual-CMU / never persisted" framing in older lessons below; `fetchCmuGraduateById` no longer resolves from a 5-min live cache (it reads the table), and `/api/cmu-alumni/sync` now MATERIALIZES (not just reports).

### CMU GET requests can't carry a body — `fetchCmuGraduateById` uses the cached list
- **Symptom:** Signup returned `503` "ไม่สามารถติดต่อระบบทะเบียนเพื่อยืนยันข้อมูลได้" whenever an applicant wasn't matched locally (so the CMU fallback ran).
- **Root cause:** `fetchFromCmuApi` always attached an auth body (`cmuaccount_name`/`api_id`) — fine for the POST list call, but `fetchCmuGraduateById` used `method: "GET"`, and Node's `fetch` (undici) throws `TypeError: Request with GET/HEAD method cannot have body`. That throw was caught in `/api/alumni-auth/signup` and surfaced as the 503. The CMU **list** (POST) worked, so the bug only appeared on the by-id path.
- **Prevention:** `fetchCmuGraduateById` (`lib/cmu-registrar.ts`) now resolves from the cached `fetchCmuGraduates()` list (FON-filtered, 5-min cache, shared with the dashboard/all-alumni/import flows) instead of a per-id GET — reliable and fast. `fetchFromCmuApi` was also fixed to pass auth as a query string (not a body) for GET. If you add another CMU endpoint, never send a body with GET/HEAD. (This also fixed the add-education CMU auto-fill preview `/api/cmu-alumni/lookup`, which uses the same function.)

### `Education` models multiple degrees; `Alumni` degree fields are a denormalized PRIMARY snapshot
- **Symptom:** Need one person to carry several FON degrees (distinct `studentId`/`degreeLevel`/`graduationYear`), viewable/editable per degree — but `studentId` is `@unique` on `Alumni` and the FK target for 6 related tables, so it can't simply become multi-valued.
- **Root cause:** `Alumni` was 1-degree-per-row. The fix is a 1:N `Education` model, BUT `Alumni.studentId` must stay unique+populated (it's the join key for Award/Association/GraduateCommittee/Potential/ModelRepresentative/AlumniAgency and the signup/import identity anchor).
- **Prevention:** `Education` (`@@unique([alumniId, degreeLevel])`, `studentId @unique`) holds every degree. `Alumni` KEEPS a denormalized **primary** snapshot (`studentId`/`degreeLevel`/`graduationYear`/`major`/`cohort`) + `primaryEducationId` pointing at the primary `Education` row. **Never edit the snapshot directly** — edit any `Education` row; the primary is re-derived and the snapshot re-synced automatically. **Primary = highest degree (auto-derived, never manual):** `recomputePrimaryEducation(alumniId, tx?, excludeId?)` (`lib/education-sync.ts`; ranks by `DEGREE_RANK` then most-recent graduation year) sets `primaryEducationId` to the highest-degree row then calls `syncPrimarySnapshot` to mirror the 5 fields, and runs after every add/edit/delete (`PUT`/`DELETE /api/educations/[id]` inside the `$transaction`; the two add routes + signup `approve` after the create). No schema field or endpoint exposes `primaryEducationId`, so it can't be pointed at a lower degree by hand. Adding a higher degree (or editing the highest one's level) re-points the snapshot `studentId`, and the 6 related tables follow via the existing `ON UPDATE CASCADE` FK. Any education may be deleted — deleting the primary reassigns to the next-highest (`excludeId` clears the FK before the delete). Backfilled via `scripts/backfill-educations.ts`. UI lives in `components/EducationSection.tsx` (badges the primary). The `dedupeCmuGraduatesByPerson` display dedup is unaffected — it's CMU-side; `Education` is local storage; the two coexist. `Education.firstName`/`lastName` carry the **name at study time** — backfilled from CMU (`name_th`/`surname_th`), not the alumni's current name (a person who changed surname keeps the study-time name on each degree card). `syncPrimarySnapshot` mirrors only the 5 degree fields, NOT the names.

### Running dev server caches the OLD Prisma client after a migration (restart; if that fails, clear `.next`)
- **Symptom A (500):** After `prisma migrate deploy` + `prisma generate`, the already-running dev server returns HTTP 500 with `PrismaClientKnownRequestError: The column alumni.<droppedCol> does not exist in the current database` — even from a plain `include: { alumni: true }` that names no columns.
- **Symptom B (404):** After `prisma generate` (regenerating `app/generated/prisma/`), the restarted dev server serves the **built-in 404** for *some* route segments only (e.g. `/login`, `/graduates/*`, `/api/alumni-profile`) while others behave normally (`/management/*` → 307 auth-redirect, `/news/*`). Restarting again does NOT fix it. The code is fine — `npm run build` + `npm run test` pass and the suspect modules load cleanly in a fresh `node --import tsx` probe — so don't chase a code bug.
- **Root cause (A):** `lib/prisma.ts` caches the client in `globalThis.__prisma` (survives HMR). The running process loaded the PRE-migration client into memory; `prisma generate` updates the files under `app/generated/prisma/` but the in-memory singleton still issues SQL for the old columns (e.g. the dropped `maidenLastName`), so every query touching the changed model 500s.
- **Root cause (B):** Turbopack's **persistent** dev cache (`.next/dev/cache/turbopack`) holds a module graph that still references the PRE-regeneration Prisma client files; a plain restart reuses it, so routes whose graph transitively imports the client fail to resolve and Next renders the built-in not-found. Routes that redirect earlier (layout/proxy auth guards → 307) never reach the broken graph, which is why the outage looks selective.
- **Prevention:** After any `prisma migrate`/`generate` against a schema change: (1) restart the dev server (Ctrl+C `npm run dev`, re-run). (2) If you still see 500s *or* selective dev-only 404s, the cache is deeper-stale — **`rm -rf .next` then restart** (a plain restart does NOT clear Turbopack's persistent cache). Always verify the data layer independently with a `node --env-file=.env --import tsx` script (uses the fresh client) and confirm `npm run build` + `npm run test` pass before blaming the code. The fresh client (build/test/tsx) is correct even while the dev server is stale. **Docker dev container (the usual setup here):** `app/generated/prisma` is an **anonymous volume** that shadows the host copy, and the running `next dev` process holds the OLD client in `globalThis.__prisma` even after the files regenerate — so a host-side `npx prisma generate` leaves the container stale (symptom: `Unknown argument \`<newField>\`` 500s). You must (a) also run `prisma generate` **inside** the container (`docker compose exec app npx prisma generate`) so its volume gets the new client, AND (b) `docker compose restart app` so the process reloads it. Generate on the host too (for `npm run lint`/`build`/`test`). For an additive nullable column the safe non-interactive flow is `migrate status` (drift check) → `migrate dev --create-only` → `migrate deploy` → `generate`; never accept a `migrate reset` on a shared DB.

- **Symptom:** A signed-up alumni "added a higher degree" by entering another person's CMU studentId; the all-alumni table still showed their old lower-degree row, and the dashboard graph mis-counted (the two different people were silently merged into one "person" under the higher degree — measured delta: Bachelor −1, total persons −1, Master unchanged — instead of the expected Master +1).
- **Root cause:** The add-/edit-education flows had no identity check, so a stranger's degree could be attached to an alumni. Per the PRD (§3.1.2) an alumni's educations are their OWN FON degrees, claimed by identity verification at sign-up. A stray degree (a) can never collapse in the all-alumni table — `dedupeCmuGraduatesByPerson` keys on name+birthday, so two different people stay as two rows (the lower-degree row persists), and (b) corrupts the dashboard — `groupPersonsByDegree` (`lib/person-degree-count.ts`) trusts that ALL of an alumni's education `studentId`s are ONE person, so the studentId bridge merges the stranger's CMU record into the alumni.
- **Prevention:** `assertEducationSamePerson` (`lib/education-identity.ts`, server-only) is called by `POST /api/alumni-profile/educations`, `POST /api/alumni/[id]/educations`, and `PUT /api/educations/[id]` (only when `studentId` changes). It is DB-only (`Education.studentId` is globally `@unique`), so it has NO fail-open: it blocks a studentId already owned by another alumni even when the birthday guard can't verify, surfacing a 'contact admin' message (admin variant names the owning alumni). The DB unique constraint + the routes' P2002 catch remain as the race/edge safety net. `excludeEducationId` skips the row being edited. The alumni-facing message is shown verbatim by `EducationSection`'s lookup preview and POST/PUT `onError`.
  - Audit/cleanup of pre-existing strays (idempotent): `node --env-file=.env --import tsx scripts/remove-stray-education.ts` (`DRY_RUN=1` to list; also deletes each stray's field-change rows, then re-syncs the name). Legitimate same-person higher degrees still merge everywhere — the CMU dedup (table) and `groupPersonsByDegree` (dashboard) already collapse a person's own multi-degree records to their highest.

### Never `git checkout <tree> -- <path>` (or `git restore --source`/`reset --hard`) with uncommitted work — it wipes it

### react-hook-form + React Compiler lint gotchas (`watch`, `Control` invariance, `zodResolver`)
- **Symptom:** Lint errors/warnings on RHF form pages: `@typescript-eslint/no-explicit-any` on `zodResolver(schema) as any`, `react-hooks/incompatible-library` ("Compilation Skipped: Use of incompatible library") wherever `watch()` is called, and — if you try to remove the `any` from a shared form helper's props — a build type error that `Control<ConcreteForm>` is not assignable to `Control<FieldValues>`.
- **Root cause:** (1) `zodResolver`'s inferred resolver type doesn't structurally match `useForm<T>`'s expected `Resolver<T>` (zod input vs output), so a cast is needed. (2) react-hook-form's `watch()` returns a value the React Compiler can't track, so it bails the whole component (`incompatible-library` is *benign* — the component still works, just isn't compiler-optimized). (3) RHF's `Control<T>` is **invariant** in `T`, so a fixed `Control<FieldValues>` rejects every concrete form type.
- **Prevention:**
  - Resolver: `resolver: zodResolver(schema) as unknown as Resolver<T>` (import `type Resolver` from `react-hook-form`).
  - `watch()`: prefer `getValues("field")` for a **one-time** read (e.g. loading an editor's initial HTML in an effect). For **reactive** `watch` (live previews/derived filters), keep it and add a documented `// eslint-disable-next-line react-hooks/incompatible-library`.
  - Shared form helpers that take `control`/`register`/`errors` (see `components/form/RepeatableFieldArray.tsx`) must be **generic** over the form's field values — `<TFieldValues extends FieldValues>(...)` with `Control<TFieldValues>` etc. — NOT a fixed `Control<FieldValues>`. The `name` prop is `FieldArrayPath<TFieldValues>` (so `useFieldArray` accepts it); dynamic per-row register paths cast `as FieldPath<TFieldValues>`; `emptyRow` stays `Record<string, unknown>` (rows use `""` for fields that coerce to number at submit). `FormInput/Select/Textarea` type their `registration` prop as `UseFormRegisterReturn` (default generic accepts any form's register return).

### Side effects inside a `setState` updater double-fire under React StrictMode (dev) — duplicated form rows
- **Symptom:** Opening an expandable section in the new-alumni form rendered that section's row **twice** (e.g. สมาคม/ชมรม showed two identical empty rows where one was expected). Only reproduced in `next dev` (StrictMode), not the production build.
- **Root cause:** `toggleSection` called `useFieldArray().append(...)` **inside** the `setSections((prev) => { … return … })` updater body. React StrictMode (on by default in `next dev`) double-invokes state-updater functions to surface impurities, so `append` fired twice → two rows. Event-handler bodies and effects are NOT double-invoked this way — only the updater function is. (Production runs once, so it looked fine there.) The `034cf9c` "one-row records" fix only touched layout/clear behavior and never this, so the duplication persisted.
- **Prevention:** Keep `setState((prev) => …)` updaters **pure** (compute + return new state only — no `append`/`remove`/`setValue`, no fetches, no external reads). Run those side effects in the event-handler body: read the current value from the render closure (`const willOpen = !sections[key]`), act on it (`if (willOpen && fields.length === 0) append(...)`), then call a pure `setSections((prev) => ({ ...prev, [key]: !prev[key] }))`. Any `setState(updater)` that does work besides returning state is a latent 2× bug in dev.

Each has `studentId String?` + `pendingStudentId String?` (+ `@@index([pendingStudentId])`); the 4 formerly-required-`studentId` models are nullable and their `alumni` relation is `Alumni?`. Their imports NO LONGER call `ensureAlumni` (no stub creation) — they use the shared `resolveAlumniLink` / `buildAlumniEntityMatchWhere` (`lib/alumni-link.ts`, server-only; `buildAlumniEntityMatchWhere` is the generalized `alumniAgencyMatchWhere`, id-or-name(id-guarded)). Dedup switched from the composite `@@unique` upsert to `findFirst({ AND: [ buildAlumniEntityMatchWhere(…), { <natural-key cols> } ] })` — once `studentId` is nullable the composite won't dedup pending rows (Postgres nulls are distinct), so the natural key (awardName+year / association+position+year / termYear+position / recordedYear / cohort+generation) is AND-ed in. Each `GET` accepts `?unlinked=true`; the page shows the amber รอเชื่อมโยง toggle + pill + effective-id cell (`studentId ?? pendingStudentId`); export round-trips `studentId || pendingStudentId`; PUT clears `pendingStudentId` when the alumni-typeahead links a real alumni. The management create forms' alumni-typeahead is still the only manual `studentId` source (the API create/update schemas' `studentId` was relaxed to optional/nullable only so pending rows can round-trip through edit — the page form still requires a pick). Import response gained `pending` + `warnings` (client-visible; the toast shows `(รอเชื่อมโยง N รายการ — ไม่มีข้อมูลศิษย์เก่า)`). `useEntityList` gained an optional `unlinked` option (conditionally keyed so non-toggle call sites don't churn); ModelRepresentative (which uses `useQuery` directly + client sort) threads `unlinked` into its own key/params and sorts by effective id. **Legacy stub `Alumni` rows from these 5 imports' pre-2026-07 behavior were reconciled 2026-07**: a one-off script removed 728 no-CMU stubs (sparse signature: `BACHELOR`+`นางสาว`+all CMU fields null + NOT in the CMU registrar) — first converting the 79 related rows pointing at them (55 awards/16 model-reps/8 agencies) to `pendingStudentId`, then deleting the stubs (cascade removed their 728 auto-`Education` rows). 4,517 sparse-but-in-CMU alumni were KEPT (real people, just under-populated — a CMU backfill is a separate task); active alumni went 11,599 → 10,871. The alumni full-form (`create-with-related`/`update-with-related`) is untouched.
- **Direct-seed gotcha (createMany/create on these 6 models omits `studentId` → silent orphan):** because `studentId` is `String?` (nullable FK), a direct-Prisma `create`/`createMany` whose `data` omits it inserts a row with `studentId = NULL` instead of erroring. Prisma still reports the created `count`, so it looks successful — but the row is **unlinked** (no `Alumni`) and invisible to every `studentId`-scoped query, `_count`, profile/API include, and the management tables. Symptom: seed prints `createMany: 3` yet `count({where:{studentId}})` returns `0` and the profile sections stay empty. Confirmed by `SELECT count(*) … WHERE "studentId" IS NULL` showing orphans. Prevention: in seed/rebuild/backfill scripts (`scripts/seed-*.ts`, `rebuild-*.ts`, `backfill-*.ts`), always spread a `base = { studentId, … }` into every row; for idempotency make the `deleteMany` clean both linked rows AND same-name NULL-`studentId` orphans (`where: { OR: [{ studentId }, { studentId: null, firstName, lastName }] }`).

### Every `buildSignupVerification` call-site must pass the local fallback (`localAuthoritativeFromAlumni`) — all 3 routes, or local-only studentIds show "ไม่พบรหัสนักศึกษา"
- **Symptom:** A REJECTED alumni re-applying with a `studentId` that exists in the local `alumni` table but NOT in `cmu_graduates` (e.g. an admin-created/legacy id like `67676767`) was shown the red "ไม่พบรหัสนักศึกษานี้ในระบบทะเบียนและไม่มีข้อมูลในระบบ" banner in the admin review modal (`VerificationFields`, `settings/users/page.tsx`) — as if no such alumni existed, even though the data was on file.
- **Root cause:** `buildSignupVerification(submitted, cmuGrad, cmuConsulted, local?)` (`lib/signup-verification.ts`) picks `source` from three branches: a CMU record → `"cmu"`, the `local` arg → `"local"`, else `null`. For a local-only studentId, CMU returns nothing WITHOUT throwing (`cmuConsulted=true`, `cmuGrad=null`), so the result is decided **entirely** by the `local` argument. The reapply route (`POST /api/alumni-auth/reapply`) was the only one of the 3 call-sites that hardcoded `null` (and never imported `localAuthoritativeFromAlumni`), so it collapsed to `source:null` → red banner. Signup and reverify already passed the local record.
- **Prevention:** There are exactly **3** `buildSignupVerification` call-sites and **all must pass the local fallback** when a local alumni record is in scope: `signup/route.ts` (`findUnique({where:{studentId}})`), `alumni-accounts/[id]/reverify/route.ts` (the in-scope `alumni` row, guarded by `stored?.source === "local"` so a freshly-created-from-submitted account never self-matches), and `reapply/route.ts` (the in-scope `alumni` row when `studentId === alumni.studentId` — a changed id is necessarily unclaimed thanks to the clash check, so `null` stays correct there). When adding a 4th caller (or refactoring), pass `localAuthoritativeFromAlumni(<alumni row>)` — **never hardcode `null`** — or local-only studentIds silently render "not found". Verified with a throwaway script against the real `67676767` row: `source` flips `null` → `"local"`.

### Drag-to-pan table scroll — one controller in the admin layout; the move threshold must gate BOTH the pan and the click
- **What:** Admins can click-and-drag any wide management table left/right to pan it horizontally (`components/table-pan/DragScrollController.tsx`, mounted once in `app/(admin)/layout.tsx`), so they don't have to scroll to the bottom to reach the horizontal scrollbar. Delegated `document` listeners auto-cover every `.overflow-x-auto` that contains a `<table>` and is actually scrollable — zero per-page edits, covers all current + future tables.
- **Rules:** mouse/trackpad only (touch scrolls natively); skip `button/a/input/select/textarea/[contenteditable]` so edit/delete/sort/links keep working; pan is horizontal only (`scrollLeft`); a genuine drag suppresses the trailing `click` (capture-phase listener) so the row's `onClick` navigation doesn't fire after a pan, but a plain click still navigates.
- **Trap — the threshold must gate the PAN, not just the click:** the 5px threshold distinguishes click vs drag. The first cut only gated the click-suppression flag and let `scrollLeft` update on every move — so a click with a few px of mouse jitter would pan slightly AND still navigate (the flag stayed false). Fix: `onPointerMove` must `return` (no `scrollLeft` write) until `|dx| > threshold`; only then set `moved=true` and pan. The `tests/drag-scroll.test.tsx` "under-threshold" case locks this in. CSS `cursor: grab` on `.overflow-x-auto:has(table)`, forced `grabbing` via a `drag-panning` class (`!important` to beat the rows' `cursor-pointer`).
- **Test setup:** DOM-touching tests need `happy-dom` (devDep) + `// @vitest-environment happy-dom` (default env is `node`). Extract side-effect logic into a plain function (`attachDragScroll(doc)`, called from the component's `useEffect`) so it can be tested without rendering React — `Object.defineProperty(el, "scrollWidth"/"clientWidth", …)` to fake scrollability, and dispatch `PointerEvent`s with `pointerType: "mouse"`.

### Expired session surfaces as a per-query "load failed" — `apiFetch` redirects 401 → `/login`
- **Symptom:** A logged-in admin/alumni whose 7-day session expired sees ONE feature fail to load — classically the alumni profile "ประวัติการเปลี่ยนแปลง" logs tab rendering `ไม่สามารถโหลดประวัติการเปลี่ยนแปลงได้` ("Failed to load…") — while the rest of the page still looks fine. Looks like a bug in that one feature (it isn't).
- **Root cause:** `proxy.ts` (Next 16 middleware) only checks that the `fon-cmu-session` cookie EXISTS, not that it's valid, so an expired cookie keeps a mounted page alive. The layout guards (`app/(admin)/layout.tsx`, `app/graduates/(authed)/layout.tsx`) re-run `getSession()` only on server navigation — a client-side tab switch / react-query refetch bypasses them. So a fresh, uncached query (the logs tab, never opened before) hits `getSession()` → expired → `null` → the route returns **401**, and the query's `isError` branch shows the generic "load failed". Cached queries (the rest of the profile) still render from the react-query cache, which is why only the one feature looked broken. **This is correct server behavior** — confirm by replaying the user's token against the route → `HTTP 401 {"error":"กรุณาเข้าสู่ระบบ"}`; a fresh session → `200`.
- **Prevention / fix:** `apiFetch` (`lib/api-client.ts`) now redirects to `${BASE_PATH}/login?expired=1` on any **401** (browser-only, with a loop guard — skip if already under `/login` — and a module-level `redirectingToLogin` flag so concurrent failing queries redirect once), then STILL throws `ApiError(401)` so react-query/callers reject cleanly while the page navigates. It does NOT redirect on **403** (logged-in-but-forbidden via `checkWritePermission` / login-email auth-state — that user is still authed). The shared `/login` page (`app/login/page.tsx`) reads `?expired=1` and shows "เซสชันหมดอายุ กรุณาเข้าสู่ระบบอีกครั้ง" (amber banner), and defaults to the **alumni** tab when `?audience=alumni` is set — `apiFetch` appends that when the expired request came from the `/graduates` area (path-side detection), so an alumni's expired session doesn't land on the admin form. **No exclusion list is needed:** the login page submits auth via raw `fetch` (never `apiFetch`), so auth-endpoint 401s (wrong password) never reach this path — every 401 through `apiFetch` is a session-expiry on a protected resource. Locked in by `tests/api-client.test.ts` (happy-dom). **Diagnostic when one feature "fails to load" but the page renders:** check the `sessions` table for a valid (`expiresAt > NOW()`) row for the user FIRST — an expired cookie is the usual cause, not the route.
- **`getSession()`/`getAlumniSession()` are admin/alumni-split:** a 401 from a `(admin)` route to an alumni session (or vice-versa) is also "wrong session type", not a bug — `getSession()` returns null for `sessionType !== "ADMIN"`, `getAlumniSession()` for `!== "ALUMNI"`. Both now funnel to the same `/login` redirect via `apiFetch`.
